# Chapter: C++ Core Concepts – Abstraction, Safety, and Expressive Power

C++ is built on the philosophy of zero-overhead abstraction. This chapter covers the foundational mechanics of the language, focusing on how we organize code, manage memory, represent data, and design object hierarchies.

## 1. Core Mechanics & Encapsulation

### Namespaces
Namespaces prevent name collisions by grouping related symbols, functions, and classes into a single semantic set. 
```cpp
namespace Engine {
    class Renderer { ... };
}
// Accessed via Engine::Renderer
```

### Classes vs. Structs
Both define custom data types. The **only** difference in C++ is default visibility:
* `class`: Members are `private` by default.
* `struct`: Members are `public` by default. (Typically used for Plain Old Data / POD).

### Anatomy of a Class
* **Member Attributes & Functions:** The data (state) and methods (behavior) belonging to the class.
* **Non-member Attributes/Functions:** Global or namespace-level variables/functions that interact with the class but aren't tied to an instance.
* **`this` pointer:** An implicit pointer passed to member functions that points to the specific instance invoking the method.
* **Accessors (Getters/Setters):** Methods used to safely read or modify private data, enforcing encapsulation.
* **Initialization Lists:** The most efficient way to initialize member variables in a constructor. *Required* for `const` members and references.
    ```cpp
    MyClass(int x) : myVar(x), myConst(10) {} 
    ```
* **`const` correctness:** Marking variables as immutable, or methods as `const` to promise they won't modify the object's state.
* **Visibility (`public`, `private`, `protected`):** Controls who can access members. `protected` is accessible to the class and its derived subclasses.
* **Pointers to Members:** Allows you to store a pointer to a specific field or method of a class, rather than a specific object instance (using `.*` or `->*`).

## 2. Memory & Object Lifecycle

In C++, an object has a strict lifecycle: from the moment memory is claimed (allocation) to the moment it is released (destruction). 

### The 4-Step Class Decoding Mental Model
When analyzing any class, look for:
1.  **DATA:** What memory/state does the object hold?
2.  **CONSTRUCTORS:** How is the object created and initialized?
3.  **METHODS:** What can the object do?
4.  **OWNERSHIP:** Who owns the memory, and who is responsible for cleaning it up?

### Stack vs. Heap Allocation
* **Stack:** Fast, automatic scoping, limited size.
* **Heap:** Slower allocation, requires manual management (or smart pointers).
    * **When to prefer the Heap:** Dynamic size objects, shared ownership, runtime polymorphism (base class pointers), and very large objects.

### RAII (Resource Acquisition Is Initialization)
The most important idiom in C++. Resources (memory, file handles, network sockets) are tied to object lifetime. Acquired in the constructor, released in the destructor.

### Smart Pointers & Move Semantics
To avoid manual `new` and `delete` memory leaks:
* `std::unique_ptr`: Exclusive ownership.
* `std::shared_ptr`: Shared ownership (reference counted).
* **Move Semantics:** Allows resources to be "stolen" from temporary objects instead of copying them, massively boosting performance.

## 3. Standard Library Staples

### `std::string`
A class managing a dynamically sized buffer of characters.
* Handles memory automatically.
* Rich manipulation methods (e.g., `.size()`, `.append()`).
* Supports operators (`+`, `==`).
* **Best Practice:** Pass as `const std::string&` to functions to avoid unnecessary deep copies.

### Range-Based For Loops
Clean syntax for iterating over containers:
```cpp
for (const auto& item : myContainer) { /* ... */ }
```

### Streams (`std::ios`)
Stream objects hold a state flag representing the health of the stream.
* `std::ios_base::iostate`: Holds error state flags.
* Functions to check state: `.good()`, `.eof()`, `.fail()`, `.bad()`.

## 4. Number Representation

### Fixed-Point Numbers
Highly applicable in environments without Floating-Point Units (FPUs), computer graphics, and sound processing. It represents real/decimal numbers using integers by pretending a decimal point exists at a fixed bit location.

* **Formula:** $\text{Real Value} = \frac{\text{Stored Integer}}{\text{Scale}}$
* **Q Format:** e.g., `Q8.8` means 8 bits for the integer part, 8 bits for the fraction. The scale is $2^8 = 256$.
* **Trade-offs:** Faster and deterministic, but you lose range, precision, and must manually manage scaling and overflow.

```c
#define SCALE 256

int to_fixed(float x) { return (int)(x * SCALE); }         // Or shift: x * (1 << 8)
float to_float(int x) { return (float)x / SCALE; }         // Or shift: x / (1 << 8)
int fixed_mul(int a, int b) { return (a * b) / SCALE; }
int fixed_div(int a, int b) { return (a * SCALE) / b; }
```

### Floating-Point Numbers
Floating point is essentially scientific notation in binary. Float stores the closest representable approximation.
* **IEEE 754 Representation:**
    $(-1)^{\text{sign}} \times (1.\text{mantissa}) \times 2^{\text{exponent} - \text{bias}}$
* The **mantissa** dictates precision; the **exponent** dictates range.

## 5. Object-Oriented Principles

### Inheritance ("IS-A" Relationship)
Inheritance creates a hierarchy of objects. It facilitates **Universal Inclusive Polymorphism**. 
* **Shadowing:** If a derived class declares a variable with the same name as a base class variable, the compiler uses the derived one. 
    * Access the base via: `BaseClass::attributeName`.
    * *Compiler flags:* `-Wshadow` warns you about this; `-Wno-shadow` suppresses it.

### Virtual Destructors (CRITICAL)
If you delete a derived object through a base-class pointer, the base class **must** have a virtual destructor. Without it, only the base class destructor fires, leaking the derived class's memory.

### Lvalues, Rvalues, and References
* **Lvalue:** An object that occupies an identifiable location in memory (has an address).
* **Rvalue:** A temporary value that does not persist beyond the expression that uses it.
* *Note on String Literals:* When you type `"I want to go home"`, it is a string literal. If a function takes `const std::string&`, the compiler takes the literal, implicitly creates a *temporary* `std::string` object (an rvalue), and binds the `const` reference to that temporary. Non-const references cannot bind to temporaries!

## 6. Two Faces of Polymorphism

In C++, objects have a **Static Type** (what the compiler sees at compile time, like a generic pointer) and a **Dynamic Type** (the actual object sitting in memory at runtime).

### 1. Runtime (Inheritance-Based) Polymorphism
Relies on a Base class, pointers/references, and `virtual` functions. Calls are resolved at runtime using a **vtable** (Virtual Table).
* **Properties:** Highly flexible (swap types at runtime), requires an inheritance hierarchy, incurs slight runtime overhead.
* **Use Case:** Common interfaces, storing heterogenous objects in a single container.
* **Abstract Class:** Contains some implementation but has at least one pure virtual function (`virtual void foo() = 0;`).
* **Interface:** A pure contract; exclusively pure virtual functions and no state.

### 2. Compile-Time (Template-Based) Polymorphism
Relies on Generic Programming. No base class or inheritance hierarchy is needed; the compiler just checks if the passed type has the required structure/functions.
* **Properties:** Static, zero runtime overhead (resolved during compilation), type-checked at compile-time.
* **Use Case:** High-performance code, STL containers, mathematical libraries.
